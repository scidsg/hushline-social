What happens to open-source projects when volunteers get busy? Maintenance piles up. Dependabot PRs stack, required upgrades accumulate, and eventually a contributor inherits days of cleanup work just to get a release moving again. We ran into this firsthand while working on hashtag#OnionShare, which ultimately led to raising the project’s first official grant to speed up the human release pipeline.

With that in mind, I’ve been playing around with a structured way to integrate AI agents (aka cron jobs for Codex) into our dev process for Hush Line, and the model is starting to feel rational.

We have two automated runners that operate under strict constraints. At midnight, the first runner audits the codebase for test coverage gaps and submits fixes for missing coverage. At 2 AM, a second runner checks whether any PRs are already open from the earlier cycle. If not, it evaluates open issues, selects the safest low-risk task, implements a fix, and opens a PR. If the results are trash, I can just close it without accepting the work.

The system is intentionally limited. The agent cannot force-push or merge its own work, and operates from a separate account with restricted permissions. Required workflows must pass, and every merge still requires human approval.

The goal is Darwinian. Even if a project temporarily loses funding or active contributors, reliability improves, tests stay maintained, and safe backlog work continues. If a human team joins or returns, nothing about their workflow changes. The agent functions as a part-time engineer focused on maintenance and stability, ensuring the project never stalls simply because resources fluctuate.

It’s also cool because, if you wanted to, you could run this on a headless Raspberry Pi and get the same results.

I think open-source projects and non-profit software services have a strategic advantage with this setup, too. All of our code is public, so it’s not like we’re dealing with trade secrets. The result is that projects stay alive.
